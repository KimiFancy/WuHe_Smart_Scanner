## 坑 1：WiFi 初始化 abort —— NVS 未初始化

### 现象
```
E (886) wifi_init: Failed to deinit Wi-Fi driver (0x3001)
ESP_ERROR_CHECK failed: esp_err_t 0x1101 (ESP_ERR_NVS_NOT_INITIALIZED)
file: "./main/internet_mgr.c" line 88
expression: esp_wifi_init(&cfg)
abort() was called
```

### 根因
`esp_wifi_init()` 内部需要 NVS 存储 RF 校准数据。原代码 `main/internet_mgr.c` 已经
`#include "nvs_flash.h"`，但从未调用 `nvs_flash_init()` —— 模板里残留的 include，
对应调用被漏掉了。

### 修复
`main/internet_mgr.c` 的 `wifi_init_sta()` 开头加入标准 ESP-IDF NVS 初始化（含 erase
容错）：
```c
esp_err_t ret = nvs_flash_init();
if (ret == ESP_ERR_NVS_NO_FREE_PAGES ||
    ret == ESP_ERR_NVS_NEW_VERSION_FOUND) {
    ESP_ERROR_CHECK(nvs_flash_erase());
    ret = nvs_flash_init();
}
ESP_ERROR_CHECK(ret);
```

### 教训
- `#include` 存在 ≠ 调用存在。检查初始化代码完整性时不能只看 include。
- WiFi/BT/PHY 三个子系统都依赖 NVS；`esp_wifi_init()` 之前必须先初始化 NVS。

## 坑 2：IDF v6 硬护栏 —— "No server verification option set"

### 现象
启用 `CONFIG_WUHE_SKIP_CERT_VALIDATE=y`（应用层旁路开关）后，TLS 握手前直接报：
```
E esp-tls-mbedtls: No server verification option set in esp_tls_cfg_t structure.
                   Check esp_tls API reference
E esp-tls-mbedtls: Failed to set client configurations, returned [0x8017]
                   (ESP_ERR_MBEDTLS_SSL_SETUP_FAILED)
```

### 根因
ESP-IDF **v6.0 加了硬护栏**（`components/esp-tls/esp_tls_mbedtls.c` ~line 990）：
```c
} else {
#ifdef CONFIG_ESP_TLS_SKIP_SERVER_CERT_VERIFY
    mbedtls_ssl_conf_authmode(&tls->conf, MBEDTLS_SSL_VERIFY_NONE);   // 真正跳过
#else
    ESP_LOGE(TAG, "No server verification option set in esp_tls_cfg_t structure...");
    return ESP_ERR_MBEDTLS_SSL_SETUP_FAILED;   // ← 0x8017
#endif
}
```

旧版 IDF 默默放行"啥都不验"，v6 直接拒绝。代码里的
`skip_cert_common_name_check = true` **名字误导**——它只关 CN/SAN 主机名比对，
**不**关证书链验证，不算"verification option"。三个标志必须同时开：

| 配置 | 作用层 |
|---|---|
| `CONFIG_WUHE_SKIP_CERT_VALIDATE=y` | 应用层：让 `wuhe_cloud.c` 走旁路分支 |
| `CONFIG_ESP_TLS_INSECURE=y` | IDF 层：解锁不安全选项组 |
| `CONFIG_ESP_TLS_SKIP_SERVER_CERT_VERIFY=y` | IDF 层：在 `esp_tls_mbedtls.c` 走 `VERIFY_NONE` |

### 修复
`sdkconfig.defaults` + `sdkconfig` 同时加这三个标志（前者持久化，后者即时生效）。

### 教训
- IDF 大版本升级时一定要看 `migration-guides/`，特别是 security 部分。
- Kconfig 名字（`skip_cert_common_name_check`）≠ 行为（不关链验证）。看 API 文档，
  不要看名字猜。
- `sdkconfig.defaults` 是 canonical 源；任何"希望跨 reconfigure 保留"的配置都要写进去。

## 坑 3：littlefs 与 lwip socket 的 FD 注册竞争

### 现象
TLS 旁路生效后，下一步报：
```
E esp_littlefs: FD 54 must be <4.
E esp-tls: [sock=54] get file flags error: Bad file number
E esp-tls: Failed to open new connection
```

### 根因
算式：`LWIP_SOCKET_OFFSET = FD_SETSIZE - CONFIG_LWIP_MAX_SOCKETS = 64 - 10 = 54`。
**Socket FD 54 是第一个合法的 socket FD。** lwip 在 `sys_init()`（由
`esp_netif_init()` 触发）里通过 `esp_vfs_register_fd_range()` 预占 `[54, 64)` 范围。

`fcntl(54)` 应该派发到 lwip，但日志显示派发到了 littlefs（其 `cache_size=4`，
54 远超），说明 `s_fd_table[54].vfs_index` 指向了 littlefs。根因是**初始化竞争**：
`app_main()` 里 `wifi_start_task()` 和 `wuhe_cloud_start()` 都是异步 task，
两者并发跑：
- `wuhe_cloud_task`：`wuhe_backup_init()` → `fopen()` 索引文件 → `register_fd()`
- `wifi_mgr_task`：`wifi_init_sta()` → `esp_netif_init()` → lwip 注册 socket 范围

 unlucky 的交错会让 littlefs 的文件 FD 操作和 lwip 的范围注册互相破坏。

### 修复
把 `wuhe_backup_init()` 从 `wuhe_cloud_task` 移出来，在 `app_main()` 里
**同步**调用，且必须**早于** `wifi_start_task()`：
```c
// main/hid_host_example.c :: app_main()
display_start_up_pagevoid(NULL);
wuhe_backup_init();      // ← 新增：同步早调，消除竞争
wifi_start_task();       // 异步
wuhe_cloud_start();      // 异步（不再二次 init backup）
```
`wuhe_cloud.c::wuhe_cloud_task()` 里原来那行 `wuhe_backup_init()` 删掉
（`s_mounted` 守卫也会兜底双调用，但删了更干净）。

### 教训
- 任何会动 VFS 全局状态（FD 表、路径前缀注册）的初始化都要**串行化**，不能跨 task 并发。
- `LWIP_SOCKET_OFFSET` 不是常量，是 `FD_SETSIZE - CONFIG_LWIP_MAX_SOCKETS` 算出来的；
  调 socket 数量配置会移动整个 socket FD 范围。
- 看到 `[sock=N]` 配上另一个子系统的 FD 范围错误，立刻想到 VFS 派发问题。
- Oracle 诊断建议"升级 joltwallet/littlefs"是不可行的——1.22.1 已是最新。
  自己读源码定位到竞争条件才是正解。

## 坑 4：协议不匹配 —— HTTPS 客户端 vs plain HTTP 服务器

### 现象
FD 问题修好后：
```
E esp-tls-mbedtls: mbedtls_ssl_handshake returned -0x7200
E esp-tls: Failed to open new connection
```

### 根因
`-0x7200` = `MBEDTLS_ERR_SSL_INVALID_RECORD`（`mbedtls/ssl.h:51`）——
"收到了无法解析为 TLS 记录的字节"。典型场景：客户端发 ClientHello，
**plain HTTP 服务器**回了 `HTTP/1.1 400 Bad Request...`，客户端拿这串字节
当 TLS 记录解析失败。

上位机是 Python（Flask 默认 dev server，端口 5000，plain HTTP），
但客户端代码硬编码 `transport_type = HTTP_TRANSPORT_OVER_SSL` + `https://` URL。

### 修复
两处对齐协议：
- `main/wuhe_cloud.h:44-45`：URL `https://` → `http://`
- `main/wuhe_cloud.c:212`：`transport_type` `HTTP_TRANSPORT_OVER_SSL`
  → `HTTP_TRANSPORT_OVER_TCP`

`wuhe_cloud.c:215-224` 的 `#ifdef CONFIG_WUHE_SKIP_CERT_VALIDATE` TLS 旁路代码
保留为死代码——切回 HTTPS 调试真实证书链路时只改 `transport_type` 即可。

### 教训
- `INVALID_RECORD (-0x7200)` ≠ 证书问题。证书问题在更早阶段（验证阶段）就拦下。
  能走到 handshake 阶段才报 `INVALID_RECORD`，基本就是协议不匹配。
- 端口是强暗示：5000 = Flask 默认 HTTP；443/8443 才是 HTTPS 常见端口。
- 调试本地服务前先 `curl http://host:port/` 验证一遍协议，省得在客户端绕远路。

```

# REMOVE before any production build.
CONFIG_WUHE_SKIP_CERT_VALIDATE=y
CONFIG_ESP_TLS_INSECURE=y
CONFIG_ESP_TLS_SKIP_SERVER_CERT_VERIFY=y
```

URL 也要从 `http://192.168.2.16:5000`（本地调试）切回
